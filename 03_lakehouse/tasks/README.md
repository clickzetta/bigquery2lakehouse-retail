# tasks/

Studio Tasks orchestration for the retail dbt pipeline.

## Directory structure

```
tasks/
├── setup.py      # Creates Studio Tasks and generates SQL files below
├── ddl/          # Generated: CREATE DYNAMIC TABLE DDL (one file per model)
└── refresh/      # Generated: REFRESH DYNAMIC TABLE commands (one file per model)
```

`ddl/` and `refresh/` are generated by `setup.py` — they are not committed to git.
Run `python tasks/setup.py` to generate them with your workspace name filled in.

## What setup.py does

1. Runs `dbt compile` to get up-to-date compiled SQL
2. Generates `ddl/<task>.sql` — `CREATE DYNAMIC TABLE <workspace>.retail.<model> AS <compiled SELECT>`
3. Generates `refresh/<task>.sql` — `REFRESH DYNAMIC TABLE <workspace>.retail.<model>`
4. Creates 7 SQL tasks in Studio under the `retail_pipeline` folder
5. Sets task content to the REFRESH command
6. Wires up task dependencies matching the dbt model DAG

## Why REFRESH, not DDL, in Studio Tasks

The DDL (CREATE DYNAMIC TABLE) is run once via `dbt run`. After that, Studio Tasks
only need to trigger `REFRESH DYNAMIC TABLE` — the table definition and SQL logic
live in the dynamic table itself, not in the task.

This mirrors the original project: Airflow + Cosmos triggered `dbt run` on demand.
Here, Studio Tasks trigger `REFRESH DYNAMIC TABLE` on demand. The dynamic tables
have no `refresh_interval` (no automatic schedule), so Studio Tasks are the sole
trigger — exactly as Airflow was in the original project.
